Sprinter · front-end domain modeling · working reference

State, Transition, Plan

A repository and its open issues are a state. Pull requests are transitions over it. Everything else is either the derivation of a transition or a plan for future ones. Two seams run through the model: which layer an object belongs to, and whether Sprinter owns it or merely references it.

Plan Workstream · Epic · dependsOn — about the future. A partial order over intended transitions.
Transition Workspace → PullRequest · Attempt · Execution · Transcript — proposed locally, applied externally. Only a merge takes effect.
State Repository @ commit · open Issues — the code host's. Sprinter can only observe it.
Registry Agent — not a layer. Global, scoped to no repository and no plan, with a lifecycle of its own. Executions reference into it.

The through-line A fact about referenced state carries the state it was observed against — and is otherwise unknown, not stale. Staleness, ownership and mergeability are three instances of the same rule.

Owned or referenced

This cuts across the layers rather than aligning with them, and it is what determines whether a screen may offer an edit at all.

Read the Transition rows carefully — the layer is split by the seam. The proposal is referenced; the derivation is owned. Sprinter's entire durable contribution to the world is the derivation: the why behind a diff that the host stores without explanation.

Two consequences follow mechanically. Referenced entities carry an observedAt; owned ones do not — a reference without a timestamp is a claim about now that cannot be backed up. And referenced entities take no optimistic updates: a local write is a request to the host that may be rejected, so the only unconditionally correct local operation is refresh.

03 · State

Repository @ refs, and the open issue set

AUTHORITY — the code host. Referenced throughout. Sprinter never writes here. Every value is a replica, and it can change while Sprinter is doing nothing at all.

The state is compound and moves atomically: a merge advances the code and removes issues from the open set in one step. Closes #12 in a pull request body is not a convenience — it is the issue-half of the transition's effect, which is why closes belongs on the PullRequest rather than being inferred.

Git already gives the state an identity we are not using. A commit is a state; a merge is a transition applied. And because a transition may target something other than the default branch, the repository's state is not one head but a map of refs.

04 · Transition

Only a merge advances the machine

AUTHORITY — split, and so is ownership. Sprinter authors transitions; the host applies them. The proposal is referenced, the derivation is owned. An agent can propose all day and the state does not move.

Three of the four events are self-loops. Everything an agent does is speculative until a merge commits it.

Attempt and Execution are different things

An Attempt is one bounded effort to resolve an Issue: picked up, worked, proposed, reviewed, and driven until the pull request merges — or given up on. An Execution is one agent's continuous run, and it yields exactly one transcript. These have different cardinalities, and collapsing them models only the simplest possible case.

The Attempt is the scope boundary, not a stage within it. Everything from triage to merge happens inside one — review and merge-verification included. How that work is orchestrated underneath is an implementation question and will not resemble a chain of skills, but the unit itself holds: one issue in, one merged pull request or one abandonment out.

An attempt may fan out — an agent delegating to a specialist, a verifier, a second opinion — and each of those is a separate run with its own transcript, its own agent revision and its own lifetime. So executions form a tree rooted at the attempt, and agentId belongs on the execution, never on the attempt.

The tree stops at the attempt boundary. Agents derive transitions; they do not plan. Sequencing an epic's issues, respecting dependsOn, deciding what runs next — that is deterministic daemon logic, not agent work, so it has no execution, no agent and no transcript. An Execution's only subject is its Attempt.

// One agent's continuous run. Exactly one transcript. Forms a tree.
Execution =
  | Running  { id, attemptId, agentId, parent: ExecutionId?,
               transcript: LiveTranscript }
  | Finished { id, attemptId, agentId, parent: ExecutionId?,
               transcript: SealedTranscript, outcome }

// One issue, picked up and driven all the way to merged — or given up.
// Running holds a workspace; the terminal variants cannot — it is gone.
// The optional PR on Running is honest: mid-flight, one may not exist yet.
Attempt =
  | Running   { id, issueId, base, root: ExecutionId,
                  workspace: Workspace,
                  pullRequest: PullRequestId? }
  | Abandoned { id, issueId, base, root: ExecutionId, reason }
  | Landed    { id, issueId, base, root: ExecutionId,
                  pullRequest: PullRequestId }   // observed Merged

// The in-flight mutation. Owned, local, uncommitted, ephemeral.
// These fields are the teardown checklist, currently maintained by hand.
Workspace = {
  id, attemptId, base: CommitSha,
  path, branch, diff,
  processes, ports, tasks
}

Three things follow. The attempt's transcript is a composition, not a thing — what a screen renders is the root execution's transcript with spawn tool-calls expanded inline into their children, which is how these are already read. Cost is a fold over the tree: Usage is per-turn on session events today, and an attempt's total is a sum across its executions — currently unaskable. And the freeze relocates without changing: live on Running, sealed on Finished, one transcript per execution, total.

The handoff — derivation and mutation freeze at different times

Sealed transcripts paired with a merged pull request are a natural unit: why it changed beside what changed, both immutable. The in-flight analog is the same two halves unfrozen — live transcripts beside the worktree, the uncommitted diff against base.

Derivation — whyMutation — whatOwnership of the mutation
workingLiveTranscriptWorkspace▪ Owned · local, uncommitted
proposedstill accumulatingPullRequest.Open▫ Referenced — the handoff
landedSealedTranscript ∗PullRequest.Merged▫ Referenced

The mutation crosses the ownership seam mid-attempt: the moment a pull request opens, the diff stops being Sprinter's and becomes the host's. The derivation does not seal then — review and fixes are still to come, and each adds executions. Only when the attempt terminates is every transcript in its tree sealed.

So the two halves freeze at different moments, and the natural unit is the attempt's terminal state: all executions sealed, and a pull request observed merged. That is Attempt.Landed — the pairing you can point at is a variant, not a join.

Modelling the in-flight half also gives teardown a home. The teardown checklist — processes, bound ports, background task ids, a launched browser, the worktree, its branches — is the untyped field list of a Workspace. As an entity, releasing it is drop the workspace rather than a procedure someone has to remember, and a leaked one becomes unrepresentable rather than merely discouraged.

Target and base are different things

A branch cannot be the base, because a mutable pointer makes staleness undecidable — you cannot compare against a thing that redefines itself. A commit cannot be the target, because this lands on main is what the pull request means and no sha expresses it.

// A transition. Endpoints are not optional — they are what makes it one.
PullRequest = {
  id, repositoryId, number, url,
  target: BranchName,       // where it is destined to apply — intent, stable
  base:   CommitSha,        // the state it was computed from — fact, immutable
  head:   CommitSha,        // the state it proposes
  state:  Open | Merged { at: CommitSha } | Closed,
  closes: IssueId[],        // the issue-half of the effect
  mergeability: Mergeability,
  observedAt
}

// Never a stored boolean. A verdict is a function of
// (transition, current target state) — so it carries that state.
Mergeability = {
  applies:  Clean | Conflicted { paths },   // git decides — cheap
  verified: Passed | Failed | NotRun,     // checks decide — expensive
  against:  CommitSha                     // what both were computed on
}

Staleness is then a comparison rather than a stored flag: tipOf(pr.target) ≠ pr.base. And a Mergeability whose against is behind that tip is unknown — the front end must render it as such, never as the last value it happened to see.

Three signals, easy to confuse

SignalWho decidesCostWhat it actually means
divergedcomparisonfreenearly always true on an active repo. Not a problem, and a UI that flags it is noise.
appliesgitcheaptextual only. Clean means almost nothing about whether the result works.
verifiedfull check on the combined stateexpensivethe only signal that gates a merge. Two transitions can both apply cleanly and still break each other.

N attempts derive transitions from the same base. At most one applies cleanly; the rest must be recomputed against the state that won. This is optimistic concurrency, and the model has no version column.

One subtlety the endpoints expose: after a rebase, the Attempt's base and the PullRequest's base diverge. The transcripts explain a derivation from c0 while the pull request now proposes from c5. The provenance link weakens without breaking, and the model should be able to say so rather than implying the transcripts still describe the diff.

05 · Plan

A partial order over intended transitions

AUTHORITY — Sprinter. Owned throughout. The only layer Sprinter fully owns. It is about the future, so it is the only layer that can be wrong without being invalid.

Workstream and Epic are neither state nor transition — they are intent. This is what dependsOn has always been: a topological constraint on the order transitions may be applied in, not a property of any issue on its own.

It also explains the flat WorkStatus on both. Plan objects have no terminal state of their own; their completion is a fold over the state layer. The enum is a cache, and caches drift.

This layer is driven deterministically. Topologically sorting an epic's issues, honouring dependsOn, choosing what dispatches next — that is daemon logic, not agent work. It has no execution, no agent revision and no transcript, and nothing here is a subject an agent operates on. Agents derive transitions; the plan layer decides which ones to ask for.

That is worth stating as a boundary rather than an implementation note: it means the plan layer needs no provenance at all. Its behaviour is reproducible from its inputs, so an audit trail here is an event log, not a transcript.

An Epic may additionally bind to a host milestone. That does not make it referenced — the Epic stays Sprinter's, and the binding is a reference it holds, which is exactly the distinction the seam draws.

06 · Registry

Agent — scoped to nothing

AUTHORITY — Sprinter. Owned, and global. Not a layer. An Agent belongs to no repository, no workstream and no attempt; it has a lifecycle of its own and can be used anywhere. Executions reference into it.

That global scope is what forces the shape. A single revision is referenced by Executions across many repositories, so mutating one would corrupt provenance everywhere at once rather than locally. Agents are therefore append-only: editing writes a new revision, and a stable name groups revisions for filtering.

It also makes deletion unrepresentable. An Agent referenced by any Execution can only be retired — never removed — or the sealed transcripts that name it point at nothing.

Anything repository-shaped about an agent is therefore derived, never stored. "Agents used in this repo" is a fold over that repo's Executions.

07 · Projections

Landing — a shape, not a thing

[SealedTranscripts, merged PullRequest] is the record you want when you ask why does this code look like this. Once the Attempt spans through to merge, that pairing has an obvious home: it is Attempt.Landed together with its execution tree. Not a cross-entity join — a variant and its children.

Which is why it should stay derived. Everything it shows already exists on the attempt, its executions and the referenced pull request; as an entity it could only disagree with its own parts, and it would need storage to hold nothing new. Being derived costs it nothing — the attempt's identity is its address, so a Landing can be linked, cited and deep-linked without being a row.

But because it is always presented as a unit, it belongs in the contract, not only in the client. The daemon composes it once; the app does not walk Attempt → Executions → Transcripts → PullRequest over four round trips to render one pane.

Domain is not contract The domain carries what the system needs; the contract carries what the interface needs. Workspace is the clean case in the other direction — an entity in the domain, but process ids and bound ports have no business crossing to the app. The Swift mirror mirrors the contract, and the contract is a projection.

The one thing that would flip Landing into an entity: if it ever needs to carry something not derivable — a human annotation, a curation flag, a note about why it mattered. Then it needs identity and storage, and this decision should be revisited rather than patched.

10 · Today

States the current model admits

Each is a rule encoded as an enum plus an optional field, or an index asserting an invariant the domain does not have. None are bugs to fix; the type allows them.

  • Issue.status = landed with Issue.pr = undefined — fixed, with no transition that fixed it
  • Job.status = completed with transcriptRef = undefined — a derivation that left no record
  • Job.status = completed while Session.status = running — two lifecycles for one thing
  • Issue.pr and Job.pr disagreeing about the same issue
  • An issue closed as wontfix indistinguishable from one closed by a merge
  • A transition with no base — unrepresentable that it is stale, which is the failure that actually costs merges
  • A referenced value with no observedAt — an unbacked claim about now, rendered as though it were live
  • UNIQUE session_job(jobId) — one agent per unit of work, false the moment a skill spawns a subagent
  • A subagent's transcript with nowhere to live, and no parent to hang from
  • A worktree, port or process outliving the attempt that opened it — teardown is a remembered procedure, not a structural guarantee
  • An in-flight diff that no screen can show, because the mutation has no model until it becomes a pull request
  • A merge that happened outside Sprinter, with nowhere to land

11 · Invariants

What the shape guarantees

  • Only a merged PullRequest advances the state. Every other event is a self-loop.
  • base is always a commit; target is always a branch. Staleness is decidable only because of the first.
  • Sprinter authors transitions and never applies them. An Attempt drives toward a merge but cannot perform one — it reaches Landed only when its PullRequest is observed Merged.
  • A merge moves both halves of the state — code and open set — atomically.
  • Every fact about referenced state carries the state it was observed against.
  • A verdict whose against is behind the target tip is unknown, not its last value.
  • Referenced entities carry observedAt; owned entities do not, and take optimistic updates.
  • Every Execution holds exactly one Transcript, in both variants. total
  • An Attempt holds one or more Executions, arranged as a tree under its root.
  • An Execution is subordinate to exactly one Attempt. Agents derive transitions and never plan; the plan layer is deterministic and has no executions at all.
  • An Attempt holds no agent and no transcript of its own. Both belong to executions.
  • Only Attempt.Running holds a Workspace. A leaked worktree is unrepresentable, not merely discouraged.
  • The mutation crosses from owned to referenced when the pull request opens; the derivation seals only when the Attempt terminates. The two do not coincide.
  • Only Attempt.Landed holds a PullRequest required, and exactly one. partial
  • An Attempt spans pickup through merge. Review and merge-verification are executions inside it, never work on a separate subject.
  • A Transcript is sealed in every terminal variant by construction, not by flag.
  • An Issue is Fixed only by reference to a PullRequest whose state is Merged.
  • A Dismissed Issue owes no PullRequest, but keeps whatever Attempts it accrued.
  • One Issue, many Attempts. Retries are new Attempts, never mutations.
  • An AgentId names an immutable revision. Editing an agent appends; it never rewrites what a past Execution ran on.
  • An Agent is scoped to nothing — no repository, no workstream. Anything repository-shaped about one is a fold over Executions.
  • An Agent referenced by any Execution can be retired but never deleted.
  • The Plan layer may be wrong about the State layer without either being invalid.

12 · Parked

None of these block the model

Each is policy or purely additive — the types are identical under every answer. They are listed so they are not forgotten, not so they are decided. The answers come from a real screen; deciding them on a whiteboard is guessing, and hardens the guess into a type.

  1. Does Sprinter keep a history of observed states, or only the latest?

    A fold implies a log. If only the head is retained, a sealed transcript will reference a base the front end can no longer resolve or diff against — a dead link in the inspector, on exactly the records meant to be permanent.

  2. Who computes verified, and when?

    It is the only signal that gates a merge and the only expensive one. Speculatively on every base move is wasteful; only at merge time means the board can never show readiness. There is a trigger policy here, and it is a product decision.

  3. When is an observedAt too old to render?

    The principle says unknown-not-stale, but every referenced value is stale the instant it arrives. Some threshold turns "recent enough to show plainly" into "show with a warning" into "refuse and refresh" — and it likely differs per entity.

  4. How deep does an execution tree go, and does the UI flatten it?

    Nesting is unbounded in principle. Whether a screen renders three levels inline or collapses everything below the first is a presentation call the model does not need to make.

  5. What happens when a bound Epic and its milestone disagree?

    The Epic is owned, the milestone is referenced, and the binding makes them describe the same intent from two authorities. Someone has to win, or the disagreement has to be first-class and visible.
